* 引言

在"Agents 安全杂谈（一）"中，我们梳理了 AI Agent 面临的威胁模型和防御思路。

本文是系列的第二篇，我们将深入四个主流 Agent 框架的内部，对比它们的安全设计：

| 框架 | 定位 | 语言 | 特点 |
|------+------+------|------|
| **OpenClaw** | 个人 AI 助手平台 | TypeScript/Node.js | 全渠道集成、技能系统、Gateway 架构 |
| **Nanobot** | 超轻量助手框架 | Python/asyncio | ~4000 行核心代码、MessageBus 解耦 |
| **Magent** | Emacs 集成编程助手 | Emacs Lisp | gptel 后端、FSM 工具调用、权限系统 |
| **Claude Code** | CLI 编程助手 | TypeScript/Node.js | 深度 Shell 集成、会话持久化 |

我们将从以下维度进行对比：
1. 权限系统设计
2. 工具调用安全
3. 会话与记忆管理
4. 沙箱与隔离机制
5. 配置安全

* 一、权限系统设计对比

**1.1 Magent：基于规则的细粒度权限**

Magent 的权限系统是最为精细的，采用 Emacs Lisp 的 alist 结构：

#+begin_src emacs-lisp
;; 默认权限规则
((read_file . ((t . allow)
               ("*.env.example" . allow)
               ("*.env" . deny)
               ("*.env.*" . deny)))
 (edit_file . ((t . allow)
               ("*.env" . deny)
               ("*.env.*" . deny)))
 (bash . ask)
 (emacs_eval . ask)
 (doom_loop . ask)
 (external_dir . ask))
#+end_src

*匹配逻辑*：
1. 精确匹配工具名 → 若有嵌套规则则进入文件匹配
2. 文件规则按声明顺序遍历，第一个 glob/精确命中立即返回
3. `*` 作为兜底（fallback）
4. 无匹配则默认 `allow`

*权限值*：
- `allow` — 直接执行
- `deny` — 拒绝，返回错误
- `ask` — 弹窗请用户确认

*优点*：
- 细粒度到文件级别
- 支持 glob 通配符
- 不同 Agent 可有独立权限配置

*局限*：
- 规则是静态的，无法适应动态上下文
- 用户需要理解 Emacs Lisp 语法
- 规则冲突难以调试

**1.2 Nanobot：配置驱动的权限模型**

Nanobot 通过配置文件定义工具白名单：

#+begin_src json
{
  "tools": {
    "allow": ["read_file", "write_file", "exec", "web_search"],
    "exec": {
      "security": "allowlist",
      "ask": "on-miss"
    }
  }
}
#+end_src

*特点*：
- 工具级别白名单
- `exec` 工具支持三种安全模式：`deny` / `allowlist` / `full`
- `ask` 参数控制是否弹窗确认

*优点*：
- 配置简单直观
- 支持环境变量覆盖（`NANOBOT_TOOLS_ALLOW`）

*局限*：
- 无文件级别细粒度控制
- 无法针对不同会话设置不同权限

**1.3 OpenClaw：Gateway 级权限控制**

OpenClaw 的权限设计分为两层：

*工具层*：
#+begin_src json
{
  "tools": {
    "allow": ["read", "web_search", "exec", "memory"],
    "exec": {
      "security": "allowlist",
      "ask": "on-miss"
    }
  }
}
#+end_src

*Gateway 层*：
#+begin_src json
{
  "gateway": {
    "nodes": {
      "denyCommands": [
        "camera.snap",
        "camera.clip",
        "screen.record",
        "contacts.add",
        "calendar.add",
        "reminders.add",
        "sms.send"
      ]
    }
  }
}
#+end_src

*特点*：
- 工具允许列表 + Gateway 命令黑名单
- 支持 `groupPolicy` 控制群组访问（`open` / `allowlist`）
- 支持 `elevated` 工具权限隔离

*优点*：
- 双层防护（工具 + Gateway）
- 群组策略灵活

*局限*：
- 默认配置偏宽松（`groupPolicy="open"`）
- 需要手动配置安全策略

**1.4 Claude Code：最小化权限哲学**

Claude Code 采用"最小权限 + 人工确认"策略：

*默认行为*：
- 读取文件：允许（工作区内）
- 写入文件：允许（工作区内）
- 执行命令：需要确认（危险命令）
- 网络访问：需要确认

*特点*：
- 工作区隔离（workspace-only 模式）
- 危险命令自动识别（`rm`、`curl | bash` 等）
- 会话级别权限继承

*优点*：
- 开箱即用的安全默认值
- 用户无需复杂配置

*局限*：
- 无法细粒度自定义
- 确认疲劳问题

**1.5 权限系统对比总结**

| 框架 | 细粒度 | 配置复杂度 | 动态调整 | 推荐场景 |
|------+--------+------------+----------+----------|
| Magent | ⭐⭐⭐⭐⭐ (文件级) | 高 | 否 | Emacs 重度用户、需要精细控制 |
| Nanobot | ⭐⭐⭐ (工具级) | 低 | 否 | 快速部署、简单场景 |
| OpenClaw | ⭐⭐⭐⭐ (双层) | 中 | 部分 | 企业部署、多用户环境 |
| Claude Code | ⭐⭐ (工作区级) | 极低 | 否 | 个人开发、快速上手 |

* 二、工具调用安全对比

**2.1 工具调用状态机（FSM）**

Magent 和 Nanobot 都采用了 FSM 来管理工具调用流程：

#+begin_src plantuml
@startuml
[*] --> INIT
INIT --> SEND : 发送 LLM 请求
SEND --> WAIT : 等待响应
WAIT --> PROCESS : 接收响应
PROCESS --> TOOL : 检测到 tool_calls?
TOOL --> CHECK_PERMISSION : 权限检查
CHECK_PERMISSION --> EXECUTE : 允许
CHECK_PERMISSION --> ERROR : 拒绝
EXECUTE --> SEND : 结果追加，继续循环
PROCESS --> DONE : 无工具调用
@enduml
#+end_src

*优点*：
- 可审计的执行轨迹
- 可在状态转换时插入安全检查
- 支持中断和恢复

*OpenClaw* 使用类似的迭代循环，但增加了 `maxIterations` 限制（默认 40 次）防止无限循环。

*Claude Code* 采用简化的请求 - 响应模式，无显式 FSM。

**2.2 工具参数验证**

| 框架 | 参数验证 | 路径遍历防护 | 命令注入防护 |
|------+----------+--------------+--------------|
| Magent | 基础类型检查 | glob 规则匹配 | 依赖权限系统 |
| Nanobot | Pydantic 验证 | 工作区限制 | security 模式 |
| OpenClaw | JSON Schema | workspaceOnly 选项 | allowlist/deny |
| Claude Code | 运行时检查 | 工作区限制 | 危险命令识别 |

**2.3 工具结果处理**

*Nanobot*：
- 工具结果截断到 500 字符
- base64 图片替换为 `[image]` 占位符
- 减少持久化存储的敏感信息

*OpenClaw*：
- 支持配置截断长度
- 敏感文件内容自动脱敏（需配置）

*Magent*：
- 完整保留工具输出
- 依赖 gptel 的流式处理

*Claude Code*：
- 完整保留输出
- 会话历史加密存储（可选）

* 三、会话与记忆管理对比

**3.1 会话隔离**

| 框架 | 会话密钥格式 | 持久化格式 | 隔离级别 |
|------+--------------+------------+----------|
| Magent | `{channel}:{chat_id}` | JSON 文件 | 渠道级 |
| Nanobot | `{channel}:{chat_id}` | JSONL 文件 | 渠道级 |
| OpenClaw | `agent:{agentId}:{channel}:{id}` | JSON | 会话级 |
| Claude Code | `{workspace}:{conversation_id}` | SQLite | 工作区级 |

*Nanobot* 的会话管理最为简洁：
#+begin_src python
# session_key 格式
session_key = f"{channel}:{chat_id}"

# 持久化
# {workspace}/sessions/{session_key}.jsonl
#+end_src

*OpenClaw* 支持更复杂的层级：
#+begin_src
agent:main:telegram:direct:7637…
agent:main:qqbot:group:group:7d…
#+end_src

**3.2 记忆系统**

*Nanobot* 两层记忆：
- `MEMORY.md` — 长期记忆（重写式合并）
- `HISTORY.md` — 历史日志（追加式）

合并触发条件：未合并消息数 ≥ `memory_window`

*OpenClaw*：
- 向量数据库 + 全文搜索（FTS）
- 支持语义检索
- 记忆分块（chunks）

*Magent*：
- 会话历史持久化
- 无独立长期记忆
- 依赖 Emacs org-mode 笔记系统

*Claude Code*：
- 项目级记忆（`.claude/memory.md`）
- 会话内上下文窗口管理
- 支持 `@` 引用文件

**3.3 记忆安全**

| 风险 | Magent | Nanobot | OpenClaw | Claude Code |
|------+--------+---------+----------+-------------|
| 记忆泄露 | 中 | 低（截断） | 中 | 低（加密可选） |
| 跨会话污染 | 低 | 低 | 低 | 低 |
| 激活值攻击 | N/A | N/A | N/A | N/A |

*注*：激活值攻击（Activation Attacks）是 2025-2026 年最新研究发现的威胁，目前主流框架均无防护。

* 四、沙箱与隔离机制对比

**4.1 执行沙箱**

| 框架 | 沙箱类型 | 配置方式 | 性能开销 |
|------+----------+----------+----------|
| Magent | 无（依赖 Emacs） | N/A | 无 |
| Nanobot | 可选（Docker） | `sandbox.mode` | 10-30% |
| OpenClaw | 可选（Docker/节点隔离） | `gateway.nodes` | 10-30% |
| Claude Code | 工作区隔离 | `--workspace` | 无 |

*OpenClaw* 的沙箱设计最为灵活：
#+begin_src json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "all",
        "workspaceOnly": true
      }
    }
  }
}
#+end_src

**4.2 网络隔离**

| 框架 | 出站限制 | 入站限制 |
|------+----------+----------|
| Magent | 无 | 无 |
| Nanobot | 可配置白名单 | 依赖 Channel |
| OpenClaw | Gateway 级控制 | 设备认证 |
| Claude Code | 用户确认 | 无 |

**4.3 TEE/机密计算支持**

目前四个框架均**无原生 TEE 支持**，但：

- *OpenClaw* 可部署在 Confidential Containers 中
- *Nanobot* 支持 Docker + TEE 组合
- *Magent* 依赖 Emacs，难以容器化
- *Claude Code* 可运行在 TEE 虚拟机中

* 五、配置安全对比

**5.1 敏感信息管理**

| 框架 | API Key 存储 | 环境变量支持 | 加密存储 |
|------+--------------+--------------+----------|
| Magent | gptel 管理 | 支持 | 否 |
| Nanobot | config.json | `NANOBOT_` 前缀 | 否 |
| OpenClaw | Gateway 配置 | 支持 | 否 |
| Claude Code | `.claude/config` | 支持 | 可选 |

**5.2 配置验证**

*Nanobot* 使用 Pydantic 进行配置验证：
#+begin_src python
class NanobotConfig(BaseSettings):
    model_config = SettingsConfigDict(
        alias_generator=to_camel,
        env_prefix='NANOBOT_'
    )
    
    tools: ToolsConfig
    agents: AgentsConfig
    # ...
#+end_src

*OpenClaw* 使用 JSON Schema 验证：
#+begin_src json
{
  "$schema": "https://openclaw.ai/config-schema.json",
  "tools": { ... },
  "gateway": { ... }
}
#+end_src

**5.3 安全配置建议**

*OpenClaw 推荐配置*：
#+begin_src json
{
  "channels": {
    "telegram": {
      "groupPolicy": "allowlist",
      "groups": ["allowed_group_id_1", "allowed_group_id_2"]
    }
  },
  "tools": {
    "exec": {
      "security": "allowlist",
      "ask": "on-miss"
    },
    "fs": {
      "workspaceOnly": true
    }
  },
  "gateway": {
    "controlUi": {
      "dangerouslyDisableDeviceAuth": false
    }
  },
  "plugins": {
    "allow": ["channels", "telegram"]
  }
}
#+end_src

* 八、选型建议

**企业部署** → **OpenClaw**
- 完整的权限和审计系统
- 支持多用户、多渠道
- Gateway 架构便于集中管理

**个人开发者（Emacs 用户）** → **Magent**
- 深度 Emacs 集成
- 细粒度权限控制
- 适合代码审查和编程任务

**快速原型/研究** → **Nanobot**
- 代码简洁（~4000 行）
- 易于理解和修改
- 适合安全研究和实验

**轻量级编程助手** → **Claude Code**
- 开箱即用
- 最小配置
- 适合个人项目开发

* 九、结语：没有银弹

通过对比四个框架，我们发现：

1. **权限细粒度** 与 **配置复杂度** 成反比
2. **沙箱安全性** 与 **性能开销** 成正比
3. **默认安全** 往往意味着 **功能限制**

安全是一场持续的博弈，没有"一次性解决"的方案。

作为用户，我们应该：
- 根据场景选择合适的框架
- 遵循最小权限原则
- 定期审查配置和日志
- 保持框架和依赖更新
